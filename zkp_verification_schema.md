# Project Astraea: Zero-Knowledge Verification Schema
## zk-SNARK Location-Tenure and Asset Privacy Specifications

This document defines the mathematical constraints, public/private input structures, and zk-SNARK circuit logic for the Project Astraea **Stateless Tokenization Proxy**. By utilizing zero-knowledge proofs (ZKPs), the system proves property occupancy, boundary compliance, and structural creditworthiness on a public Layer-2 ledger without exposing precise GPS coordinates or visual data.

---

## 1. ZK-Relation Formalization

Let the relation $R$ define the criteria for a valid occupancy and structural creditworthiness proof:

$$R = \{ (x, w) \mid \text{VerifyGeohash}(w_{gps}, x_{h3}) \land \text{VerifyFootprint}(w_{poly}, w_{area}) \land \text{VerifyOwnership}(w_{key}, x_{addr}) \land \text{VerifyScore}(w_{metrics}, x_{acs}) \}$$

Where:
*   $x$ is the vector of **public inputs** exposed on the L2 ledger.
*   $w$ is the vector of **private inputs (witnesses)** kept confidential.

## 2. Input Parameter Schemas

To represent float-based spatial coordinate systems inside prime-field cryptography (such as the BN254 curve used in Groth16 zk-SNARKs), coordinates are scaled to 32-bit integers:

$$\text{ScaledVal} = \text{FloatVal} \times 10^7$$

### A. Private Inputs (Witness $w$)
These values are hashed locally on the edge client or stateless proxy. They are never transmitted to the blockchain.

| Parameter | Type | Representation / Constraints | Description |
| :--- | :--- | :--- | :--- |
| `w_lat_centroid` | `uint32` | Latitude $\times 10^7$ (e.g., `-13129480` for `-1.312948`) | Exact coordinate latitude centroid. |
| `w_lon_centroid` | `uint32` | Longitude $\times 10^7$ (e.g., `367902140` for `36.790214`) | Exact coordinate longitude centroid. |
| `w_boundary_vertices` | `uint32[8]` | $4 \times (\text{lat}, \text{lon})$ pairs scaled by $10^7$ | Polygon boundary vertices of the physical footprint. |
| `w_occupant_privkey` | `field element` | 256-bit private scalar | Claimant's private signature key. |
| `w_raw_image_hash` | `bytes32` | Poseidon Hash of raw binary smartphone images | Cryptographic identifier linking the proof to physical photos. |
| `w_structural_metrics` | `uint16[4]` | `[BDI, RoofingMaterial, WallMaterial, EnvScore]` | Raw inputs from the VLM structural analytics engine. |

### B. Public Inputs (Exposed on-chain $x$)
These values are committed to the public Layer-2 state contract upon successful proof verification.

| Parameter | Type | Representation | Description |
| :--- | :--- | :--- | :--- |
| `x_h3_index` | `uint64` | H3 Index at Resolution 12 (approx 0.1m precision) | Mapped structural sector index cell. |
| `x_occupant_address` | `address` | Standard 20-byte EVM address | The public key authorized to manage the tokenized asset. |
| `x_asset_credit_score`| `uint16` | Normalized value: `100` to `900` | The calculated Asset Credit Score. |
| `x_oracle_signature` | `bytes` | ECDSA signature of the Astraea verification node | Proves the VLM processed the images matched to `w_raw_image_hash`. |

## 3. Circuit Constraint Logic (ZK-SNARK)

Astraea utilizes a zero-knowledge circuit (implemented in DSLs such as Noir or Circom) enforcing the following constraints:

### Constraint 1: Geo-Spatial Index Containment
Proves that the private coordinate centroid $(w_{\text{lat}}, w_{\text{lon}})$ lies precisely within the bounds of the public H3 hexagonal index cell $x_{\text{h3\_index}}$:

$$\text{H3CentroidDistance}\left((w_{\text{lat}}, w_{\text{lon}}), \text{H3ToCentroid}(x_{\text{h3\_index}})\right) \le R_{\text{hex}}$$

Where:
*   `H3ToCentroid` is an in-circuit implementation of the H3 index projection algorithm.
*   $R_{\text{hex}}$ is the bounding radius of a resolution 12 hexagon (approx. $0.6$ meters).

### Constraint 2: Footprint Area Coherence
Proves that the surface area calculated from the boundary vertices $w_{\text{poly}}$ corresponds to the area metrics used in structural evaluation:

$$\text{CalculatePolygonArea}(w_{\text{boundary\_vertices}}) == w_{\text{footprint\_sqm}}$$

This uses the Shoelace Algorithm configured within prime field arithmetic constraints:

$$2 \times \text{Area} = \left| \sum_{i=1}^{n-1} (x_i y_{i+1} - x_{i+1} y_i) + (x_n y_1 - x_1 y_n) \right|$$

### Constraint 3: Identity and Key-Ownership Verification
Proves that the occupant address $x_{\text{occupant\_address}}$ matches the public key generated by the private key witness $w_{\text{occupant\_privkey}}$, without revealing the private key:

$$x_{\text{occupant\_address}} == \text{PubKeyDerivation}(w_{\text{occupant\_privkey}})$$

### Constraint 4: Algorithmic Score Bounds Verification
Proves that the public $x_{\text{asset\_credit\_score}}$ matches the deterministic calculation performed on the private structural metrics $w_{\text{structural\_metrics}}$:

$$x_{\text{asset\_credit\_score}} == w_{\text{calculated\_acs}}$$

Where the circuit checks:
1.  `BDI` score verification: Checks that the VLM-extracted building degradation index is valid ($0.0 \le \text{BDI} \le 1.0$).
2.  `ACS` formulation matching: Checks that the linear scoring formula is executed correctly within the circuit parameters.

## 4. Ingress Routing flow with ZKP

```
[Smartphone / Field App]
    |-- Coordinates & Photos Captured
    |-- Poseidon Hash Generated (w_raw_image_hash)
    v
[Edge / Prover Engine]
    |-- Computes zk-SNARK Proof (π)
    |-- Emits Public Inputs (x) & Witnesses (w)
    v
[Stateless Ingress Proxy]
    |-- Verifies Oracle Signature
    |-- Routes Proof (π) and Public Inputs (x) to L2
    v
[Arbitrum / Polygon L2 Verifier Contract]
    |-- Executes: VerifyProof(π, x)
    |-- Mints Deed NFT (mapping TokenID -> x_h3_index + x_asset_credit_score)
```

This verification schema guarantees that an informal property owner can prove to any micro-lending institution that they occupy a valid, non-overlapping structure with a verified structural rating, while keeping their exact geographical boundaries completely private on-chain.
